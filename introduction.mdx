Watchexec is a simple, standalone tool that monitors a directory and runs a command whenever it detects file modifications. This documentation covers everything from installing the binary to filtering events and consuming the JSON event stream.

## What watchexec does

When you run `watchexec make`, it executes `make` immediately, then watches the current directory recursively. When any file changes, it runs `make` again. Multiple rapid filesystem events — like the swap files that editors write during a save — are coalesced into a single run so your command doesn't fire a dozen times for one logical change.

Watchexec reads `.gitignore`, `.ignore`, and other VCS ignore files automatically, so build artifacts and version-control internals are excluded without any configuration. It uses kernel-level APIs (inotify on Linux, FSEvents on macOS, ReadDirectoryChangesW on Windows) so it doesn't burn CPU in a polling loop.

## Key features

**Event coalescing** — Multiple filesystem events generated by a single logical change (editor swap files, atomic writes) are debounced into one, controlled by the `--debounce` flag (default: 50 ms).

**Gitignore and ignore file support** — Watchexec automatically loads `.gitignore`, `.ignore`, and global ignore files. It detects which VCS is in use and loads only the relevant files.

**Process groups and job objects** — Commands run in a process group (Unix) or Job Object (Windows) so that signals and terminations reach the entire process tree, not just the top-level shell. This matters when your command forks child processes.

**Event emission** — Changed file paths are passed to your command via environment variables (`$WATCHEXEC_WRITTEN_PATH`, etc.) or as a newline-delimited JSON stream on stdin with `--emit-events-to=json-stdio`. Use `--only-emit-events` to consume the event stream without running any command.

**Busy-update control** — When a new event arrives while a command is still running, you choose what happens: ignore it (`do-nothing`, the default), queue another run (`queue`), kill and restart (`restart`, or `-r`), or send a signal (`signal`).

**Cross-platform, no runtime** — A single static binary works on Linux, macOS, Windows, and BSD. There is no language runtime or ecosystem dependency.

<Note>
Watchexec is not tied to any particular language or build system. It runs shell commands, so it works with `make`, `cargo`, `npm`, `go build`, or anything else you can invoke from a terminal.
</Note>

## Filtering

You can narrow which file changes trigger the command using several independent mechanisms, applied in order:

- **`--fs-events`** — filter by event kind (create, remove, rename, modify, metadata)
- **`-w` / `--watch`** — limit to specific files or directories
- **Ignore files** — `.gitignore`, `.ignore`, and others loaded automatically
- **`-i` / `--ignore`** — explicit glob patterns to exclude
- **`-e` / `--exts`** — quick extension filter (e.g. `-e js,ts,css`)
- **`-f` / `--filter`** — explicit glob patterns to include
- **`-j` / `--filter-prog`** — custom jaq programs for advanced logic

<Tip>
Use `--print-events` to see which events are reaching the command after filtering. Add `-vvv` for full diagnostic output.
</Tip>

## Rust library

If you need to embed file-watching behaviour in your own Rust tool, the [`watchexec` library crate](https://crates.io/crates/watchexec) exposes the same engine as the CLI: event sources, filterers, process supervision, and an action handler API.

Related crates built alongside watchexec:

- [`watchexec-events`](https://crates.io/crates/watchexec-events) — event types
- [`watchexec-signals`](https://crates.io/crates/watchexec-signals) — signal types
- [`watchexec-supervisor`](https://crates.io/crates/watchexec-supervisor) — process lifecycle management
- [`ignore-files`](https://crates.io/crates/ignore-files) — find, parse, and interpret ignore files

Notable tools built on the watchexec library include [cargo-watch](https://github.com/watchexec/cargo-watch) and [ghciwatch](https://github.com/mercurytechnologies/ghciwatch).
